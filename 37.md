AWS Infrastructure Labs: Day 37 - IAM Instance Profile & S3 Integration
📋 Task Overview
The Nautilus DevOps team needs to set up an application on an EC2 instance to interact with a private S3 bucket for secure data storage and retrieval. This hands-on lab demonstrates how to provision a private Amazon S3 bucket, enforce granular Identity and Access Management (IAM) least-privilege policies, map an IAM instance profile to a running EC2 node, and validate the end-to-end data pipeline using the AWS Command Line Interface.

Objectives
Automated Key Exchange: Create an RSA identification key pair on the management platform (aws-client) and distribute the public key to the xfusion-ec2 node for seamless passwordless terminal handshakes.

Provision Private Object Storage: Create a fully isolated, private Amazon S3 bucket designated as xfusion-s3-274233372499.

Enforce Least-Privilege IAM Access:

Author a custom IAM policy granting explicit permissions only for s3:PutObject, s3:ListBucket, and s3:GetObject against the targeted bucket resource.

Attach the policy definition to a new execution role named xfusion-role and assign it as an Instance Profile to xfusion-ec2.

Pipeline Verification: Connect to the instance via SSH and execute object staging and query commands to confirm data compliance.

🛠️ Step-by-Step Implementation Guide
Step 1: Configure Network Security Groups & Password-less Key-Based Authentication
Log into your aws-client administration terminal host to build and sync the secure key pair architecture.

Bash
# 1. Generate an identity key matrix if it does not exist on your host
if [ ! -f /root/.ssh/id_rsa ]; then
    ssh-keygen -t rsa -N "" -f /root/.ssh/id_rsa
fi

# 2. Extract and display the compiled public key string
cat /root/.ssh/id_rsa.pub
Copy the string output.

Pivot to the AWS Management Console and open a terminal interface onto your xfusion-ec2 instance using EC2 Instance Connect or Session Manager (acting as the root administrative operator).

Inject the authorization string into the target server profile:

Bash
mkdir -p /root/.ssh
chmod 700 /root/.ssh
echo "PASTE_THE_COPIED_PUBLIC_KEY_STRING_HERE" >> /root/.ssh/authorized_keys
chmod 600 /root/.ssh/authorized_keys
Step 2: Create a Private S3 Bucket
Navigate to the AWS Console -> S3 Dashboard -> click Create bucket.

Configure Bucket Properties:

Bucket name: xfusion-s3-274233372499

AWS Region: Select the default region assigned to your lab environment.

Under Block Public Access settings for this bucket:

Ensure Block all public access is checked (This enforces bucket isolation and privacy).

Leave all other parameters (Bucket Versioning, Encryption) at their factory default values.

Scroll to the bottom and click Create bucket.

Step 3: Author the Custom IAM Policy and Role
Part A: Build the Granular IAM Policy
Navigate to the IAM Dashboard -> click Policies in the left navigation sidebar -> click Create policy.

Select the JSON editor tab and completely replace the boilerplate statement with the following least-privilege specification block:

JSON
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "VisualEditor0",
            "Effect": "Allow",
            "Action": [
                "s3:PutObject",
                "s3:GetObject"
            ],
            "Resource": "arn:aws:s3:::xfusion-s3-274233372499/*"
        },
        {
            "Sid": "VisualEditor1",
            "Effect": "Allow",
            "Action": "s3:ListBucket",
            "Resource": "arn:aws:s3:::xfusion-s3-274233372499"
        }
    ]
}
Click Next.

Policy name: xfusion-s3-access-policy (or any preferred descriptive name).

Click Create policy.

Part B: Create the IAM Role (xfusion-role)
In the IAM sidebar, click Roles -> click Create role.

Select trusted entity: Choose AWS service -> from the service dropdown, select Lambda or EC2 (Select EC2 as this will be used as an instance profile). Click Next.

Add permissions: Search for your newly created policy xfusion-s3-access-policy, select its checkbox, and click Next.

Role name: Type exactly xfusion-role.

Review the structure mappings and click Create role.

Step 4: Attach the IAM Instance Profile to xfusion-ec2
Navigate back to the EC2 Dashboard -> click Instances.

Select the checkbox next to the xfusion-ec2 instance.

Click the Actions dropdown menu at the top right -> select Security -> click Modify IAM role.

From the IAM role dropdown list, choose xfusion-role.

Click Update IAM role to attach the instance profile configuration.

🧪 Verification & End-to-End Validation
Log into your aws-client manager engine space to connect directly to the application host via your distributed SSH keys and perform the pipeline storage test. Replace <XFUSION_EC2_PRIVATE_IP> with the internal private IP address of your application instance.

Bash
# 1. Access the application server via passwordless SSH
ssh root@<XFUSION_EC2_PRIVATE_IP>

# 2. Generate an arbitrary local mockup file to stage for transfer
echo "Testing serverless object store pipeline integration" > metadata-test.txt

# 3. Synchronize the asset up to your private S3 storage bucket
aws s3 cp metadata-test.txt s3://xfusion-s3-274233372499/

# 4. Inquire against the target bucket registry to confirm transmission success
aws s3 ls s3://xfusion-s3-274233372499/
Expected Outcome

Your terminal screen interface will report back a clean completion metadata confirmation string displaying the object details matching the format below:

Plaintext
2026-06-18 14:15:32         53 metadata-test.txt
This output registers that your granular IAM instance profile authentication loop and private object storage gateway pipeline are completely operational.
