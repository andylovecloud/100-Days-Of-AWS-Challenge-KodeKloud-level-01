# AWS Serverless Labs: Day 34 - Deploying Lambda via AWS CLI

## 📋 Task Overview
The Nautilus DevOps team is expanding its adoption of serverless architectures by establishing deployment pipelines via the command line interface. The goal of this challenge is to author, package, and deploy an AWS Lambda function using the AWS CLI rather than the management console, reinforcing automation practices.

### Objectives
1. **Create Python Script:** Develop a Python script named `lambda_function.py` that returns a status code `200` and the body text `"Welcome to KKE AWS Labs!"`.
2. **Package the Code:** Compress the script into a deployment package named `function.zip`.
3. **Provision the Function:** Deploy a new Lambda function named `xfusion-lambda-cli` via the AWS CLI using the `python` runtime.
4. **Access Management:** Attach the pre-configured IAM role named `lambda_execution_role`.

---

## 🛠️ Step-by-Step Implementation Guide

Follow these sequential terminal commands on your `aws-client` host management platform to complete the lab.

### Step 1: Create the Workspace and Python Script
Establish a isolated working directory and write the compliant runtime execution code.


# Create and navigate to a new workspace directory
mkdir -p ~/lambda-cli-lab && cd ~/lambda-cli-lab

# Write the Lambda execution logic into lambda_function.py
```bash
cat << 'EOF' > lambda_function.py
import json

def lambda_handler(event, context):
    return {
        'statusCode': 200,
        'body': json.dumps('Welcome to KKE AWS Labs!')
    }
EOF
```

Step 2: Package the Code into a Zip Archive
AWS Lambda expects code deployments to be compressed. Compress the file using the local zip utility.

```
Bash
zip function.zip lambda_function.py
Verification: You can run ls -l to ensure function.zip exists in your active directory.
```

Step 3: Fetch the IAM Execution Role ARN
To map permissions via the CLI, you must locate the explicit Amazon Resource Name (ARN) bound to the execution profile.

```
Bash
aws iam get-role --role-name lambda_execution_role --query "Role.Arn" --output text
```
Copy the output string (e.g., arn:aws:iam::123456789012:role/lambda_execution_role) for the next step.

Step 4: Deploy the Function to AWS Lambda
Execute the deployment creation string. Replace <YOUR_ROLE_ARN> with the identifier retrieved in Step 3.

```
Bash
aws lambda create-function \
    --function-name xfusion-lambda-cli \
    --runtime python3.12 \
    --role <YOUR_ROLE_ARN> \
    --handler lambda_function.lambda_handler \
    --zip-file fileb://function.zip
```
⚠️ Note on File Prefixes: The prefix fileb:// is a strict parameter requirement in AWS CLI that forces the environment to ingest the archive stream as binary data rather than plain text.

🧪 Verification & Testing
Validate that the runtime executes successfully and yields the targeted output payload. Depending on your system configuration, use the appropriate syntax block below.

Option A: For AWS CLI Version 1 (Standard Core)

```
Bash
aws lambda invoke \
    --function-name xfusion-lambda-cli \
    --payload '{}' \
    response.json
```

Option B: For AWS CLI Version 2 (Latest Release)

```
Bash
aws lambda invoke \
    --function-name xfusion-lambda-cli \
    --cli-binary-format raw-in-base64-out \
    --payload '{}' \
    response.json
```

Inspecting the Payload Results
Print the output file generated by the execution layer to confirm validation requirements match:

```
Bash
cat response.json
Expected Output
JSON
{"statusCode": 200, "body": "\"Welcome to KKE AWS Labs!\""}
```

A successful status code response from the endpoint indicates that the infrastructure pipeline is fully operational.
